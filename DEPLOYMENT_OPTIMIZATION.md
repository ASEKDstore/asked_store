# Deployment Optimization - Render.com

## Текущая ситуация

### Проблема производительности

При текущей конфигурации каждый сервис в Render выполняет:
- `npm ci` - установка всех зависимостей монорепозитория
- `npm run build:<service>` - сборка конкретного сервиса

Это означает:
- **Backend**: `npm ci` + `npm run build:api`
- **Frontend**: `npm ci` + `npm run build:webapp`
- **Bot**: `npm ci` + `npm run build:bot`

**Итого**: `npm ci` выполняется 3 раза при каждом деплое, что значительно замедляет процесс.

### Оценка времени

- `npm ci` для монорепозитория: ~30-60 секунд (в зависимости от количества зависимостей)
- Сборка сервиса: ~10-30 секунд
- **Общее время деплоя**: ~2-4 минуты минимум

При частых деплоях это критично влияет на скорость доставки изменений.

## Варианты оптимизации

### Вариант 1: Разделение репозитория на сервисные репо

**Подход**: Разделить монорепозиторий на отдельные репозитории:
- `asked-store-api`
- `asked-store-webapp`
- `asked-store-bot`
- `asked-store-shared` (npm package)

**Плюсы**:
- Каждый сервис устанавливает только свои зависимости
- Быстрая сборка каждого сервиса (~30-60 секунд вместо 2-4 минут)
- Независимые деплои (деплой одного сервиса не влияет на другие)
- Более простые CI/CD пайплайны

**Минусы**:
- Потеря единой точки управления версиями
- Сложность синхронизации изменений между репо
- Необходимость публиковать `shared` как npm package
- Сложность совместной разработки (нужны PR в нескольких репо)
- Потеря возможности делать atomic changes (изменения в API + Frontend одновременно)

**Рекомендация**: ❌ Не рекомендуется для текущей стадии проекта. Усложняет разработку без достаточной выгоды.

---

### Вариант 2: Использование build cache в Render

**Подход**: Использовать Render's build cache для `node_modules`

**Реализация**:
- Render автоматически кеширует `node_modules` между билдами
- Использовать `cacheDirectories` в `render.yaml` (если поддерживается)

**Плюсы**:
- Минимальные изменения в конфигурации
- Ускорение `npm ci` за счёт кеша (с ~60s до ~10-20s)
- Работает "из коробки" в Render

**Минусы**:
- Render может очищать кеш при изменении зависимостей
- Не решает проблему полностью (всё равно нужен `npm ci`)

**Рекомендация**: ✅ Использовать как первую оптимизацию. Проверить наличие автоматического кеширования в Render.

---

### Вариант 3: Оптимизация монорепозитория (selective installs)

**Подход**: Использовать `npm ci --workspace` для установки зависимостей только нужного workspace

**Реализация**:
```yaml
# Backend
buildCommand: npm ci --workspace=@asked-store/api && npm run build:api

# Frontend
buildCommand: npm ci --workspace=@asked-store/webapp && npm run build:webapp

# Bot
buildCommand: npm ci --workspace=@asked-store/bot && npm run build:bot
```

**Плюсы**:
- Установка только необходимых зависимостей
- Быстрее чем полный `npm ci`
- Сохраняется структура монорепозитория

**Минусы**:
- `npm ci --workspace` всё равно устанавливает зависимости всех workspace (из-за hoisting)
- Не даёт значительного ускорения
- Может вызывать проблемы с зависимостями между workspace

**Рекомендация**: ⚠️ Проверить возможность, но вероятно не даст значительного эффекта.

---

### Вариант 4: Docker-based builds с multi-stage

**Подход**: Использовать Docker с кешированием слоёв зависимостей

**Реализация**:
- Создать Dockerfile с multi-stage build
- Кешировать слой с `node_modules`
- Render поддерживает Docker-based деплой

**Плюсы**:
- Полный контроль над процессом сборки
- Эффективное кеширование слоёв Docker
- Возможность оптимизировать каждый шаг

**Минусы**:
- Необходимость поддерживать Dockerfiles
- Усложнение конфигурации деплоя
- Render может иметь ограничения на Docker builds

**Рекомендация**: ⚠️ Рассмотреть в будущем, если другие варианты не дадут результата.

---

### Вариант 5: Использование pnpm вместо npm

**Подход**: Заменить npm на pnpm для лучшего кеширования и производительности

**Плюсы**:
- pnpm значительно быстрее npm
- Лучшее использование дискового пространства (hard links)
- Встроенное кеширование
- Поддержка workspaces

**Минусы**:
- Необходимость обновить все скрипты
- Проверить совместимость с Render
- Нужно обучить команду

**Рекомендация**: ✅ Рассмотреть как потенциальную оптимизацию после проверки поддержки Render.

## Рекомендуемый план действий

### Фаза 1: Текущее состояние (принято)
- ✅ Оставить текущую конфигурацию
- ✅ Документировать проблему
- ✅ Мониторить время сборки

### Фаза 2: Быстрые оптимизации (когда появится время)
1. Проверить автоматическое кеширование Render
2. Оптимизировать зависимости (удалить неиспользуемые)
3. Рассмотреть переход на pnpm (если поддерживается)

### Фаза 3: Среднесрочные оптимизации (при необходимости)
1. Реализовать selective installs (если это даст эффект)
2. Оптимизировать структуру workspace для минимальных зависимостей

### Фаза 4: Долгосрочные оптимизации (только при критической необходимости)
1. Разделение репозитория (только если станет критичным)
2. Docker-based builds (если другие варианты не работают)

## Метрики для мониторинга

- Время сборки каждого сервиса
- Время установки зависимостей (`npm ci`)
- Частота деплоев
- Количество зависимостей в корневом `package.json`

## Технический долг

- **Приоритет**: Низкий (не блокирует разработку)
- **Влияние**: Замедление деплоя на 1-2 минуты
- **Решение**: Отложено до момента, когда это станет критичным

## Связанные документы

- `render.yaml` - текущая конфигурация деплоя
- `package.json` - структура монорепозитория
- `ARCHITECTURE.md` - архитектурные принципы
